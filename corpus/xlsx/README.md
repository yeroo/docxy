# xlsx conformance corpus

Workbooks whose formula cells carry **cached values computed by an engine
other than gridcore**, used as an oracle: `gridcore` recalculates every
formula and diffs against the embedded values. Run the scoreboard on any
workbook (or several — an aggregate is printed) with:

```sh
xlsxy corpus/xlsx/*.xlsx --verify
```

Current scoreboard: **461/461 formula cells (100%)** across 17 files.

## The test suite over this corpus (runs in CI)

- `gridcore/tests/conformance.rs` — recalculate every file, zero mismatches
  allowed against the embedded oracle values.
- `gridcore/tests/roundtrip.rs` — four invariants per file:
  - load → save → reload is semantically identical, and a second save is
    byte-identical (deterministic writer);
  - parts we don't model survive byte-for-byte;
  - every cell renders (number formats, dates, errors) and exports to CSV
    without panicking;
  - recalculation is idempotent (a second full recalc changes nothing).

## Files

- **`oracle-basic.xlsx`** — 42 formulas across the general surface. Oracle:
  the Python [`formulas`](https://pypi.org/project/formulas/) engine, with
  one correction where that engine deviates from Excel (`TRIM` must collapse
  internal space runs — D34).
- **`calc-*.xlsx`, `shape-*.xlsx`, `edge-cases.xlsx`** — generated by
  `corpus/tools/gen_xlsx_corpus.py`: themed workbooks authored with openpyxl
  (formulas only), then recalculated by **LibreOffice Calc headless**, which
  embeds its computed values. Areas: arithmetic/operators, math, statistics,
  text, logic, lookup, dates, financial, criteria (`*IF`/`*IFS` with
  wildcards), cross-sheet + whole-column + defined-name references, a
  realistic amortization table (exercises shared-formula groups as written
  by LibreOffice), a gradebook (INDEX/MATCH, nested IF), a real Excel Table
  with calculated columns and structured references
  (`shape-salestable.xlsx`), 3D sheet spans (`calc-3d.xlsx`), a TEXT()
  battery across the format-code surface (`calc-textfmt.xlsx`), and edge
  cases
  (XML-hostile text, unicode, long dependency chains, error values, merged
  cells).

To regenerate (requires `libreoffice-calc` and python3 + openpyxl):

```sh
python3 corpus/tools/gen_xlsx_corpus.py
```

## Oracle caveats

- Post-2007 function names must be written with their `_xlfn.` prefix in
  stored formulas, or engines cache `#NAME?` (the generator handles this).
- LibreOffice 24.2 predates `XLOOKUP` (added in 24.8) and the dynamic-array
  functions (`FILTER`/`SORT`/`UNIQUE`/`SEQUENCE`…), so it can't oracle those;
  they are covered by gridcore's unit tests until a real-Excel corpus file
  (or a newer LibreOffice) provides oracle values.
- LibreOffice does not evaluate the bare `[@Col]` sugar when authored via
  openpyxl (it caches `#N/A`); the equivalent explicit
  `Table[[#This Row],[Col]]` form works and is what the generator emits.
  gridcore supports both (unit-tested).
- Known LibreOffice-vs-Excel deviations in this corpus: none currently.
  Document any future ones here, with the corrected cell noted.

## Adding corpus files

The best oracle is **real Excel**: open a workbook in Excel, let it
calculate, save — the cached values in the file are the ground truth. Drop
the file in this directory; the CI suite picks it up automatically. Keep
workbooks small and focused, and prefer values that don't depend on locale,
timezone, or the current date (`--verify` excludes volatile formulas
automatically).
