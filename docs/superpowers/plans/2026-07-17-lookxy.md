# lookxy Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A terminal (TUI) mail client that reads and triages corporate Microsoft 365 mail (Exchange Online) over the Microsoft Graph API, syncing to a local SQLite store for offline use.

**Architecture:** Two workspace crates following the house pattern. `mailcore` is a headless engine — hand-rolled JSON, OAuth2 authorization-code + PKCE auth (system browser + loopback redirect), a `ureq`+`rustls` Graph REST client, a `rusqlite` (bundled, FTS5) store, and a channel-driven background sync engine. `lookxy` is the ratatui/crossterm TUI over the synced store. The store is the single source of truth for the UI; all network work happens on the sync thread.

**Tech Stack:** Rust (edition 2024), ratatui/crossterm, ureq + rustls, rusqlite (bundled SQLite + FTS5), windows-sys (DPAPI). Hand-rolled JSON, OAuth, Graph protocol, and sync.

## Global Constraints

- Edition 2024; MSRV 1.88 (`edition.workspace = true`, `rust-version.workspace = true`).
- New workspace members added to root `Cargo.toml` `members`: `mailcore`, `lookxy`.
- `[lints] workspace = true` in every new crate; `cargo clippy -D warnings` must stay green.
- Inline `#[cfg(test)]` module tests (house convention — no separate `tests/` dir unless a fixture corpus is needed).
- No async runtime. Blocking I/O on a dedicated sync thread; UI and sync communicate over `std::sync::mpsc` channels.
- Hand-roll JSON, OAuth, and the Graph protocol. Only TLS (`rustls` via `ureq`) and the SQLite store (`rusqlite`) are third-party — this is the deliberate, spec-sanctioned break from pure-std cores.
- Auth flow: **authorization-code + PKCE with a `http://localhost:<port>` loopback redirect** (interactive system browser). Device-code flow is NOT used — EPAM Conditional Access blocks it (validated in Task 0).
- OAuth client ID: **`14d82eec-204b-4c2f-b7e8-296a70dab67e`** (Microsoft Graph CLI public client), validated against the EPAM tenant by the auth spike. Tenant endpoint: `https://login.microsoftonline.com/organizations/…` at runtime (per-user tenant), `epam.com` confirmed working. Scopes: `Mail.ReadWrite offline_access`. The ID is a config constant, overridable via `LOOKXY_CLIENT_ID`.
- Secrets never logged. Token cache DPAPI-encrypted on Windows.
- Graph base: `https://graph.microsoft.com/v1.0`. Honor `Retry-After` on 429/503; never tight-loop.

### Task 4: OAuth2 authorization-code + PKCE auth (`mailcore::auth`)

**Files:**
- Create: `mailcore/src/auth.rs`
- Create: `mailcore/src/pkce.rs` (PKCE + URL helpers, unit-testable in isolation)

**Rationale:** Device-code flow is blocked by EPAM Conditional Access (Task 0). The validated flow is authorization-code + PKCE with a system-browser sign-in and a loopback (`http://localhost:<port>`) redirect. Reference: `scratchpad/spike-pkce.ps1`.

**Interfaces:**
- Consumes: `crate::json`, `crate::pkce`, `crate::testserver` (tests).
- Produces (`pkce.rs`):
  - `struct Pkce { pub verifier: String, pub challenge: String }` and `fn Pkce::generate() -> Pkce` — 32 random bytes → base64url (no pad) verifier; challenge = base64url(sha256(verifier)). SHA-256 is hand-rolled (small, std-only, unit-tested against known vectors) so no crypto dependency enters the tree.
  - `fn base64url_encode(bytes: &[u8]) -> String` / `fn base64url_decode(s: &str) -> Option<Vec<u8>>`.
  - `fn form_urlencode(pairs: &[(&str, &str)]) -> String`.
- Produces (`auth.rs`):
  - `struct AuthConfig { pub authority: String, pub client_id: String, pub scope: String }` with `fn default() -> AuthConfig` using the validated client ID + `https://login.microsoftonline.com/organizations`.
  - `struct TokenSet { pub access_token: String, pub refresh_token: String, pub expires_at_unix: u64, pub account: String }`
  - `struct AuthRequest { pub authorize_url: String, pub redirect_uri: String, pub state: String, pkce: Pkce }` — everything needed to drive the browser + catch the redirect.
  - `fn begin_auth(cfg: &AuthConfig, redirect_uri: &str) -> AuthRequest` — builds the `/authorize` URL (`response_type=code`, `redirect_uri`, `response_mode=query`, `scope`, `state`, `code_challenge`, `code_challenge_method=S256`, `prompt=select_account`).
  - `fn redeem_code(cfg: &AuthConfig, http_base: &str, req: &AuthRequest, code: &str) -> Result<TokenSet, AuthError>` — POSTs `grant_type=authorization_code` with `code`, `redirect_uri`, `code_verifier` to `{http_base}/{tenant}/oauth2/v2.0/token`.
  - `fn refresh(cfg: &AuthConfig, http_base: &str, refresh_token: &str) -> Result<TokenSet, AuthError>` — POSTs `grant_type=refresh_token`.
  - `enum AuthError { Denied(String), Http(String), Parse(String) }`
  - `http_base` is normally `https://login.microsoftonline.com`; tests pass the fake server base. Tenant = last path segment of `cfg.authority` (`rsplit('/').next()`).
  - `account` is derived from the `id_token`'s `preferred_username` claim (base64url-decode the middle JWT segment, parse JSON) or falls back to `""`.
- The loopback listener itself lives in the sync layer / a small `loopback` helper (Task 11 / Task 17 drive it); `auth.rs` stays pure request/response so it is fully fake-server-testable. The loopback helper: `fn listen_for_code(redirect_uri: &str, expected_state: &str) -> Result<String, AuthError>` — binds the `localhost:<port>` from `redirect_uri`, accepts one request, parses `?code`/`?error`/`state` from the request line, writes a tiny "you can close this tab" HTML page, returns the code (verifying `state`).

**Files:**

**Interfaces:**

  - `enum SyncEvent { FoldersUpdated, MessagesUpdated{folder_id:String}, BodyReady{id:String}, State(SyncState), Error(String), SignInRequired, SignInStarted{authorize_url:String} }` — `SignInRequired` means "no valid token, user must trigger sign-in"; `SignInStarted` carries the authorize URL the UI opens in the browser.

Then enumerate folders (emit `FoldersUpdated`), backfill each folder's delta within `backfill_days`, then loop: handle commands (translating mutations into optimistic store writes + `enqueue_op`), drain outbox, and every 60s re-run deltas. Token 401 anywhere → refresh once → retry; refresh failure → emit `SignInRequired`. Network errors → `Offline` state + back-off.
